### F-200 (P0) — `pre_phase3_readout_scanner` silently missing weekly runs since 2026-04-27

**Evidence**
- Live registry: `pre_phase3_readout_scanner` `status='operational'`, `cadence='weekly'`, `last_run_utc=2026-04-27 12:00:30`.
- `scanner_runs` table contains exactly 2 rows for this scanner since 2026-04-27, both on 2026-04-27 (one at 10:26 emitting 115 signals, one at 12:00 emitting 0).
- `dispatch_weekly` cron is `0 12 * * 1` (Mondays 12 UTC). Mondays since 2026-04-27 = 2026-05-04 and 2026-05-11. `takeover_candidate_scanner` ran on 2026-05-04 12:00 (proves dispatch_weekly fired), but `pre_phase3` did not.
- `_load_cadence_names("weekly", _DEFAULT_SCANNERS_WEEKLY)` (`modal_workers/app.py:702`) calls `load_operational_names_by_cadence` (`modal_workers/shared/supabase_client.py:247`), which filters `cadence=eq.weekly + status=eq.operational`. Live registry satisfies both for `pre_phase3` → registry should return it.
- Therefore: either the registry GET succeeded but excluded `pre_phase3` for an unidentified reason (e.g. transient PostgREST inconsistency), OR `_dispatch` looked up `getattr(me, "pre_phase3_readout_scanner_once", None)` and got `None` (function not deployed at the time), OR the spawn raised silently.

**Impact**
- A whole class of biotech catalysts (Phase-3 readout watchlist; 115 signals per cold run) has been missing from the funnel for 2 weeks.
- Per memory `email_alert_gating.md`, Immediate-band emails depend on the funnel — silently missing scanner = silently missing alerts.

**File:line**
- `modal_workers/app.py:857-864` (`dispatch_weekly`)
- `modal_workers/app.py:702-716` (`_load_cadence_names`)
- `modal_workers/app.py:614-616` (`pre_phase3_readout_scanner_once`)
- `modal_workers/shared/supabase_client.py:247-261` (`load_operational_names_by_cadence`)

**Recommendation**
1. Manually trigger `pre_phase3_readout_scanner_once` to backfill (signals are time-sensitive — readouts in the next 90d).
2. Pull last 30 days of `dispatch_weekly` Modal logs (`modal app logs conan-v2 --function dispatch_weekly`) to find the actual cause for the 2026-05-04 miss.
3. Add an observability sweep that flags `operational + cadence != on_demand` scanners whose `last_run_utc` is older than `2× cadence period`. (Today, `edgar_runtime_health` is the only per-scanner liveness check and it's EDGAR-specific.) See F-204.
4. Consider hardening `_dispatch` to write a row into `operator_flags` (severity=warn) whenever `errors[]` is non-empty or `spawned` is shorter than the requested name list — currently the envelope only goes to Modal logs.

**Follow-up:** spawn-task

**Evidence**

- `dispatch_release_times` reads `load_operational_daily_names_for_hour(6)` from registry → empty for this scanner → never spawned. `openfda_corpus_ingest_once` is on-demand-only as a result.

**Impact**
- v3 RAG corpus is not being refreshed daily as designed. Any downstream skill that joins `documents` against `fda_assets` is reading a corpus that was last ingested via manual trigger or never.
- Sunday 180d deep sweep is also dead (Sunday auto-trigger lives in scanner code, but scanner itself never runs).

**File:line**
- `supabase/migrations/20260507120000_add_openfda_corpus_ingest_scanner.sql` (entire file)
- `modal_workers/app.py:619-628` (function exists; comment misleading)

**Recommendation**
1. Apply the migration immediately (manual via Supabase MCP `apply_migration`, or via the supabase CLI). Reuse the file as-is — `ON CONFLICT (name) DO NOTHING` is safe.
2. Investigate why this migration alone was missed during the 2026-05-07 deploy wave (10+ migrations applied that day).
3. After applying, manually trigger `openfda_corpus_ingest_once` once to seed the corpus, then verify the next 06 UTC `dispatch_release_times` tick fires it.
4. Long term: a CI check that asserts every `INSERT INTO public.scanners` migration has a matching `*_once` function in `app.py` and vice-versa would catch this drift.

**Follow-up:** spawn-task

### F-204 (P1) — No "stale operational scanner" detector

**Evidence**
- `dispatch_observability` runs 6 sweeps every 6h (`scanner_probe`, `edgar_runtime_health`, `orphan_convergence_sweeper`, `provisional_convergence_audit`, `thesis_jobs_sla_sweeper`, `pre_edge_monitor`).
- `edgar_runtime_health` is the only per-scanner liveness check and it's EDGAR-specific.
- `scanner_failure_streak` flags consecutive failures only — not silent absence (no runs at all).
- Result: F-200 went undetected for 14 days.

**Impact**
- Any operational scanner can stop running silently for arbitrary intervals; the only way to notice is manually inspecting `scanners.last_run_utc` columns.

**File:line**
- `modal_workers/app.py:886-1000` (`dispatch_observability`)
- `modal_workers/observability.py:1252-1341` (`edgar_runtime_health` — model for the new sweep)

**Recommendation**
- Add a new sweep `scanner_liveness_sweep` invoked from `dispatch_observability`. For each `status='operational'` scanner with `cadence != 'on_demand'`, compute `expected_max_age_s` from cadence (3h=10800, daily=86400, weekly=604800) and emit `operator_flags(severity='warn', source='scanner_liveness', kind='stale_run', metadata={scanner, last_run_utc, expected_max_age_s, age_s})` when `now() - last_run_utc > 2 × expected_max_age_s`.
