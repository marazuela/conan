### F-001: Watch→active promotion (60d catalyst rule) is enforced by a Claude skill, not by code
- **Severity:** P1
- **Component:** candidate_aging
- **Tier:** T1
- **File:line:** [.claude/skills/candidate_aging.md:56-99](.claude/skills/candidate_aging.md)
- **Evidence:** `.claude/skills/candidate_aging.md:56` defines: "`state='watch'` AND `extensions->>'routine_declined' IS DISTINCT FROM 'true'` AND catalyst lands in the next 60 days". Lines 74-79 define aged-out watch (60d), 82-92 define stale-active demote (30d + no catalyst within 60d). No equivalent SQL trigger, RPC, or scheduled SQL job exists — confirmed by grepping `60` / `catalyst` / `promote` across `supabase/migrations/`, `supabase/functions/`, `modal_workers/`. The only Modal-side actor (`pre_edge_monitor.py`) handles only `binary_catalyst` and `takeover_candidate` deterministic transitions; it does **not** implement the 60d rule.
- **Hypothesis:** Memory entry [candidate_watch_active_promotion.md](memory) is correct, but the rule is enforced exclusively by Claude reading the skill in a Cowork scheduled task. There is no test coverage, no CI guard, and no automated regression detection if the skill drifts. A subtle change in the skill text (e.g., dropping the daterange/date arithmetic comment at line 60-62 about `daterange && tstzrange` failing silently) could silently disable the entire promotion stage and the system would continue running without alerting.
- **Suggested fix:** Either (a) port the deterministic Stage A predicates to a SQL function or scheduled job so they are unit-testable, or (b) add a lightweight integration test that fires the skill against a fixture DB and asserts the expected transitions. Maintain operator_flags coverage for skill-execution failures.
- **Disposition:** spawn-task

### F-002: Thesis_writer 15/day cap is enforced inside the Claude skill via an SQL count, not a DB constraint
- **Severity:** P1
- **Component:** thesis_writer
- **Tier:** T1
- **File:line:** [.claude/skills/thesis_writer.md:5,60-70](.claude/skills/thesis_writer.md)
- **Evidence:** Line 5 documents "quota: 15 promotions per UTC day (soft cap...)". Lines 60-70 give the SQL count query the skill must run before drafting. Lines 72-77 define the `short_positioning` sub-quota (default 5/day). No `CHECK` constraint, no SQL trigger, and no Postgres function enforces the cap; thesis_jobs schema (migration 20260420200000) has no daily-count guard.
- **Hypothesis:** If a future operator runs `thesis_writer` from a different session, or if the skill is edited to remove the count query, the cap is silently bypassed. The cap exists to protect alert volume; bypass = email flood.
- **Suggested fix:** Add a Postgres function or trigger that enforces "no more than 15 promotions per UTC day" at the candidate_events INSERT (or candidates UPDATE) layer. Skill check becomes a courtesy / fast-fail; DB is the actual gate.
- **Disposition:** spawn-task
