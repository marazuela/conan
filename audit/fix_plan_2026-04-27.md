## Design constraints (read these before any fix)

These constraints are deduced from `spec.md` and the existing architecture. They override naive "fix everything" instincts.

1. **Skill-resident logic is intentional** (per [spec.md §7.4 line 812](spec.md), [§7.5](spec.md)). The 60-day catalyst rule, 15/day thesis cap, and Stage A aging predicates live in `.claude/skills/*.md` because Cowork-scheduled Claude execution is the chosen runtime. **Do NOT port these to SQL triggers.** The optimal fix for skill-resident invariants is *integration tests* + *drift detection*, not relocation.
2. **`internal_config` table is the runtime-mutable-config surface.** Any threshold that should be tweakable without redeploy goes here, not in code. Already used for compute_secret + Modal URLs. Existing pattern: service-role-only access, key/value/updated_at schema.
3. **`candidate_transition_apply` RPC is the only valid path to mutate `candidates.state`.** Per migration 20260426000000. Enforce by trigger; don't audit by convention.
4. **Modal endpoint contracts are guarded by `CONAN_COMPUTE_SECRET`.** All five compute endpoints (rescore_with_dims, assess_thesis, render_candidate_markdown, regex_check, storage_upload) already enforce. The reactor's `rubric_apply_caps` is the public exception by design (called by Supabase webhook, no skill auth).
5. **Severity-ordered, dependency-aware execution.** Some clusters unblock others — e.g., the shared pagination helper (Cluster C) is a prerequisite for migrating any new scanner. Execute in the order below.

# C1 — Pagination silent-loss [P0] — **narrowed 2026-04-27**

**Reality check (2026-04-27 execution):** F-112 was hallucinated (asx_scanner doesn't paginate; it rotates a static ticker universe). F-108 in `congressional_trading.py` is the only real P0 pagination bug. Two new findings (F-121 courtlistener single-page, F-122 EDGAR `[:50]` truncation) are P2 and **deferred** — both volume-bounded, no immediate loss.

Per CLAUDE.md ("three similar lines is better than a premature abstraction"), abandoning the shared-helper plan. With one real caller, a targeted fix is the optimal answer; abstraction comes when there's a second.

**Problem (root cause):** [congressional_trading.py::_fetch_trades](modal_workers/scanners/congressional_trading.py:343) treats every break-condition the same — transient fetch error, parse failure, no-table-on-page, no-rows-on-page all `break` immediately. The dangerous case is "no-table" or "fetch error" mid-loop, since Capitol Trades pages are publication-date sorted and a transient blip on page 3 silently drops pages 4..N.

**Optimal fix:** Distinguish three break classes and handle each correctly:
1. **Transient fetch/parse error** → retry once with backoff; if still failing, warn-and-break (consistent with current pessimistic stop, but observable).
2. **No `<table>` on page** → likely bot-protection or HTML schema change. Warn-and-break (continuing won't recover).
3. **`<table>` exists but no rows** → most-likely legitimate EOF for Capitol Trades, but tolerate one empty-streak before stopping to absorb the rare transient.

**Plan:**
1. Add a single retry pass to the fetch+parse phase in [congressional_trading.py:357-368](modal_workers/scanners/congressional_trading.py:357). On `requests` exception or HTML parse exception: log+sleep(2s)+retry; on second failure, append warning + break.
2. Replace `if not rows: break` (line 376-377) with empty-streak counter; break after 2 consecutive empties (default tolerant value).
3. Keep `if not table: break` semantics but ensure it appends to `warnings` with both page number and a hint to check for bot protection.
4. Tests in new file [modal_workers/tests/test_congressional_trading.py](modal_workers/tests/test_congressional_trading.py): full-page → empty-page (legit EOF), full → empty → full (sparse middle page, scanner continues), full → fetch-fail → recovers via retry, full → fetch-fail twice → break-with-warning, full → no-table → break-with-warning.
5. Note F-121 and F-122 in the audit doc (already done) for a future volume-driven follow-up.

**Closes:** F-108 (P0). F-112 rejected. F-121, F-122 deferred (logged for future).

**Verification:** New test suite. Manual replay of last 30d via the existing scanner-runs path is not part of this fix (would require live Capitol Trades access); instead, the test fixtures cover the failure modes the bug enables.

**Optimal fix (architectural):** A single reactor reliability PR that:
1. Hardens the Modal endpoint contract (env-required URL, retries, observability).
2. Aligns the unknown-profile contract on the Python side.
3. Awaits all flag writes so DLQ/operator_flags don't race the response.

**Plan:**

The current fallback `https://marazuela--rubric-apply-caps.modal.run` is a personal-namespace URL — production should never silently use it. Set the env var explicitly via Supabase Dashboard → Edge Functions → Secrets before deploy.
2. **F-203 — add retry+backoff on `rubricApplyCaps`.** Wrap the `fetch` at [reactor/index.ts:456](supabase/functions/reactor/index.ts:456) with: timeout (10s), exponential backoff (250ms / 1s / 4s), max 3 attempts, only retry on 5xx + network. Log each retry to `failed_reactor_events.payload.retries` for observability. After exhausted retries → DLQ as today.
3. **F-204 — make `score_signal` raise on unknown profile.** [rubric_engine.py:379-381](modal_workers/shared/rubric_engine.py:379) currently silently falls back to `activist_governance`. Change to raise `UnknownScoringProfile` (already defined at [rubric_engine.py:418](modal_workers/shared/rubric_engine.py:418)). Audit all callers — at minimum scanner_base needs to surface the exception as a per-signal error (`per_signal_errors` append) rather than crash the run. **Migration risk:** any scanner currently emitting an unknown profile would start failing visibly; check scanner_runs.errors over last 7 days first to confirm no in-the-wild unknowns.
4. **F-214 — `await flagHeuristicMissingScoringMeta`.** [reactor/index.ts:189-203](supabase/functions/reactor/index.ts:189) — change from fire-and-forget to awaited; the operator flag must land before reactor returns.
5. **F-203 supporting telemetry.** Add structured timing to the reactor response object: `rubric_apply_caps_ms`, `rubric_apply_caps_retries`, `convergence_classify_ms`. Surfaces latency regressions before they become DLQ floods.

**Closes:** F-202, F-203, F-204, F-214. Closes prior-audit Risk #1 (resilience portion) and Risk #3 (unknown-profile asymmetry).

# C3 — State-transition invariant [P1]

**Problem (root cause):** Migration 20260426000000 introduced `candidate_transition_apply` as the sole atomic path for mutating `candidates.state`. Currently no code bypasses it — but nothing prevents future code, skill, or operator action from doing so. F-215 calls this out: invariant is by-convention, not by-DB.

**Optimal fix (architectural):** A Postgres trigger that blocks any `UPDATE candidates SET state` outside the RPC's transaction. Pure DB-level enforcement; cannot be bypassed by application drift.

**Plan:**
1. New migration `supabase/migrations/202604300000_candidate_state_transition_guard.sql`:
   ```sql
   CREATE OR REPLACE FUNCTION public.enforce_candidate_state_via_rpc()
   RETURNS trigger LANGUAGE plpgsql AS $$
   DECLARE
     v_caller text := current_setting('conan.in_transition_rpc', true);
   BEGIN
     IF NEW.state IS DISTINCT FROM OLD.state THEN
       IF v_caller IS NULL OR v_caller != 'true' THEN
         RAISE EXCEPTION
           'candidates.state changes must go through public.candidate_transition_apply';
       END IF;
     END IF;
     RETURN NEW;
   END $$;

   CREATE TRIGGER candidate_state_transition_guard
     BEFORE UPDATE ON public.candidates
     FOR EACH ROW
     EXECUTE FUNCTION public.enforce_candidate_state_via_rpc();
   ```
2. Update [migrations/20260426000000_candidate_transition_rpc.sql](supabase/migrations/20260426000000_candidate_transition_rpc.sql) → add `PERFORM set_config('conan.in_transition_rpc', 'true', true);` at the top of the RPC body so the trigger lets the RPC through.
3. Tests: integration test asserting (a) RPC succeeds, (b) raw `UPDATE candidates SET state='killed'` raises, (c) `UPDATE candidates SET ticker='X'` succeeds (non-state mutations untouched).

**Closes:** F-215.

**Verification:** Run new migration + RPC in dev; smoke-test with both paths. Existing pre_edge_monitor + candidate_aging tests should still pass since both go through the RPC.

# C4 — Observability sweep coverage [P1/P2]

**Problem (root cause):** The 6h dispatch_observability cycle covers convergence, scanner endpoint health, thesis_jobs SLA, and litigation baselines. But four classes of silent failure aren't swept: DLQ growth, stuck `needs_scoring`, zero-emit scanner runs, and orphan-state invariants. The 2026-04-22 audit's Risk #5 (reactive-not-proactive) is partly about coverage.

**Optimal fix (architectural):** Four new sweep functions in [modal_workers/observability.py](modal_workers/observability.py), each following the existing `_upsert_flag` / `_resolve_flag` pattern. All run inside the existing `dispatch_observability` 6h cron — no new schedule needed.

**Plan:**
1. **F-308 — `failed_reactor_events_sweeper`.** Count rows added in last 24h. If >100 → P1 operator_flag. Also flag any single-event row older than 30d (long-stuck DLQ). Resolves to the table itself; do not auto-replay (replay is an explicit operator action).
2. **F-216 — `needs_scoring_sla_sweeper`.** `WHERE status='needs_scoring' AND created_at < now() - interval '7 days'`. Flag with severity scaled to count. Catches paused signal_resolver skill / API outage.
3. **F-317 — `scanner_emission_zero_sweeper`.** For each scanner, compute 30d median emissions/run. If most-recent run has `signals_emitted_count=0` and median ≥10 → operator_flag. Catches upstream HTML/API format changes that yield 0 results without a clear failure.
4. **F-307 — `state_consistency_sweeper`.** Three orphan checks in one query:
   - `candidates.state='delivered' AND no row in candidate_outcomes` — at most 100 sample
   - `candidate_events.candidate_id NOT IN (SELECT id FROM candidates)` — orphan events
   - `signals.convergence_bonus > 0 AND no winner row stamped` — inverse convergence orphan
   Each gets a distinct `kind=` so dashboards can cluster.
5. **F-306 fanout email failures** — existing path uses Resend; check whether webhook delivery is logged anywhere. If not, add a `fanout_email_failures` table + sweep on growth >5/day. (May be deferrable if Resend dashboard is the existing source-of-truth; document the decision.)

**Closes:** F-216, F-306 (with a possible deferral note), F-307, F-308, F-317.

**Verification:** Each sweep gets a unit test in [tests/test_observability.py](modal_workers/tests/test_observability.py) using fixture rows that should and shouldn't trigger flags. Run dispatch_observability against a clone of prod for one cycle; review which flags fire.

# C5 — Skill-resident invariant testing [P1]

**Problem (root cause):** Per spec.md §7.4, skills are the authoritative implementation of thesis_writer, candidate_aging, and signal_resolver. This is intentional. But there's no automated test that the skill's documented predicates (60d catalyst, 15/day cap, kill_condition regex check, etc.) actually fire correctly when Claude runs the skill against a fixture DB. A subtle skill edit could silently disable a predicate.

**Optimal fix (architectural):** Add a "skill harness" test pattern — fixture DB seeded with synthetic candidates/signals, a programmatic Claude invocation of the skill, and assertions on the resulting DB state. This is NOT a port back to SQL/Python; it preserves the design intent while regaining test coverage.

**Plan:**
1. **F-001 — `candidate_aging` integration test.** New file `modal_workers/tests/integration/test_candidate_aging_skill.py`. Fixture seeds:
   - One candidate with `state='watch'` and `next_catalyst_date = now() + 30d` (should promote)
   - One candidate with `state='watch'` and `next_catalyst_date = now() + 90d` (should NOT promote)
   - One candidate with `state='watch'`, `routine_declined=true` (should NOT promote)
   - One candidate with `state='active'` and `updated_at > 30d ago` and no near catalyst (should demote)
   - One candidate with `state='watch'` and `updated_at > 60d ago` (should age out)
   Run the skill via the existing Cowork harness (or via Anthropic API in test env). Assert resulting state transitions and `candidate_events` rows.
2. **F-002 — `thesis_writer` integration test.** Fixture seeds 20 `thesis_jobs.status='queued'` rows with realistic signal payloads. Run skill. Assert:
   - Exactly 15 promotions (or fewer if rows fail challenger).
   - Remaining 5+ stay `queued` with `gate_reasons=['daily_quota_reached']`.
   - `routine_declined=true` rows promote to `watch`-flagged without consuming quota.
3. **Drift watchdog (lightweight, observability-side):** A weekly observability sweep `skill_invariant_audit` that:
   - Counts last-7d candidate transitions per stage. If Stage A `promote_watch_active` count drops to 0 while watch population > 10, flag (skill regression suspected).
   - Counts last-7d `thesis_writer` promotions. If sustained > 15/UTC-day for 3 consecutive days, flag (cap not enforced).
4. **Spec annotation:** add a top-of-skill comment block in each Cowork skill: `# Tested by: modal_workers/tests/integration/test_<skill>.py — keep predicates in sync.`

**Closes:** F-001, F-002. Honors the spec.md §7.4 design intent.

**Verification:** Tests run in CI with a Supabase test project (or local supabase + mocked Claude). Watchdog tested by deliberately seeding a contradiction.

# C6 — Silent error-handling standardization [P1/P2/P3]

**Problem (root cause):** `except Exception: pass` appears across at least 8 scanner sites. Each was a defensible local choice ("don't crash on a transient API hiccup"); collectively they form a class of unobservable degradations. The pattern is the bug, not the individual sites.

**Optimal fix (architectural):** A shared helper in [scanner_base.py](modal_workers/shared/scanner_base.py) that takes a callable + a context label and: runs it, on exception logs structured error + appends to `per_signal_errors`/`run_metrics` via the existing path (lines 364, 384, 415). All scanner silent-pass sites migrate to this helper.

**Plan:**
1. New helper signature in scanner_base:
   ```
   def best_effort(
     fn: Callable[[], T],
     *,
     context: str,
     run_metrics: dict,
     default: T = None,
     log_level: str = "warning",
   ) -> T
   ```
   On exception: log with stack, increment `run_metrics["best_effort_failures"][context]`, return `default`.
2. Migrate every `except Exception: pass` site found in audit T2:
   - F-100 [_courtconnect_client.py:505](modal_workers/scanners/_courtconnect_client.py:505) (session close)
   - F-106 [fda_pdufa_pipeline.py:385](modal_workers/scanners/fda_pdufa_pipeline.py:385) (network error)
   - F-107 [insider_form4_scanner.py:738](modal_workers/scanners/insider_form4_scanner.py:738) (snapshot enrichment)
   - F-109 [insider_form4_scanner.py:683](modal_workers/scanners/insider_form4_scanner.py:683) (FIGI resolution)
   - F-113 [esma_short_scanner.py:524](modal_workers/scanners/esma_short_scanner.py:524) (cache parse) — handled by C10 too
   - F-115 [edgar_filing_monitor.py:51](modal_workers/scanners/edgar_filing_monitor.py:51) (retry config) — C12
   - F-116 [edgar_filing_monitor.py:56](modal_workers/scanners/edgar_filing_monitor.py:56) (market cap floor) — C12
   - F-120 [courtlistener_scanner.py:374](modal_workers/scanners/courtlistener_scanner.py:374) (OpenFIGI)
3. Existing observability `scanner_failure_streak_observer` already reads `scanner_runs.errors` — it now sees `best_effort_failures` aggregates and can flag scanners with rising rates without per-site instrumentation work.
4. Lint rule (lightweight): a `tests/test_no_silent_excepts.py` that grep-asserts no new `except Exception: pass` lands in `modal_workers/scanners/`. Failing test = code review prompt.

**Closes:** F-100, F-106, F-107, F-109, F-113 (partly), F-115 (partly), F-116 (partly), F-120.

**Verification:** New helper has its own test (success / exception / counter increment). Each migrated scanner's existing test suite continues to pass; no new failure modes introduced.

# C9 — JSONB schema enforcement [P2]

**Problem (root cause):** Several JSONB columns (`signals.raw_payload`, `signals.extensions`, `thesis_jobs.dossier`, `candidate_events.payload`, `candidates.kill_conditions`) accept arbitrary shapes. Existing validators (`validate_scoring_meta`, `_validate_structured_kill`) cover some but not all. F-208 (fanout thesis-payload) and F-302 (writer-side validation gaps) are the most exposed.

**Optimal fix (architectural):** Define a single shared schema set with two parallel implementations — Pydantic (Python writers) and Zod (TS writers/readers). Apply at every writer boundary. NOT a Postgres CHECK on JSONB shape (too rigid; future evolution would require migrations).

**Plan:**
1. **Author schemas in `modal_workers/shared/schemas.py` (Pydantic)** for: `RawPayloadSignals` (per-profile variants), `ScoringMeta`, `KillCondition`, `CandidateEventPayload`, `ThesisOutput`. Author Zod equivalents in `supabase/functions/_shared/schemas.ts`.
2. **Apply at writer boundaries:**
   - `scanner_base._signal_to_row` validates `raw_payload` and `extensions.scoring_meta` (extends existing validator).
   - `app.py::storage_upload_endpoint` validates `dossier` shape before allowing upload.
   - `fanout/index.ts:380` (F-208) validates `payload.thesis` against ThesisOutput before rendering.
   - `candidate_transition_apply` RPC validates `p_payload` shape against CandidateEventPayload (via a Postgres function calling a JSON Schema validator extension if available, or reject in skill writers).
3. **Failure mode:** validation failures are *not* hard errors. Each writer logs + flags `operator_flags(kind='schema_violation')` and proceeds with best-known shape. This catches drift without blocking the pipeline.
4. **Test corpus:** for each schema, fixture set of 20 valid + 5 invalid samples. Lock the schema by making the fixtures part of CI.

**Closes:** F-208, F-302.

**Verification:** Sample 100 random `signals.raw_payload` rows from prod through the new validator; record violations. Aim for <1% violation rate post-fix; remaining violations reveal real scanner bugs.

Each function tested.

**Plan:**

   - F-113: corrupt cache load → log + return empty state, flag `operator_flags(kind='esma_dedup_cache_corrupt')`

# C11 — Auto-cap + dim_estimator hardening [P1/P2/P3]

**Problem (root cause):** [rubric_engine.py:200-350](modal_workers/shared/rubric_engine.py) auto-caps mix early-return and append patterns inconsistently (F-206), silently archive on missing universe match (F-211), terminal-discard on definitive_merger flag without recovery (F-212), apply uniform liquidity tier across profiles (F-213), and use string-literal cap rule IDs (F-210). dim_estimator defaults missing dims to 3 uniformly (F-205). Several of these are inter-related.

**Optimal fix (architectural):** Two refactors landed together:
1. **Always-append cap evaluation.** Every cap appends to `caps_triggered`; the band falls to the lowest cap-imposed band (most restrictive wins). Predictable; no order-dependence.
2. **Per-profile dim defaults.** Replace global "missing dim → 3" with a `PROFILE_DIM_DEFAULTS` table that names a default per (profile, dim). `requires_resolution=true` is set in lockstep on any default-fired dim.

**Plan:**
1. Refactor [rubric_engine.apply_auto_caps](modal_workers/shared/rubric_engine.py:256) to always-append. Compute final band as `min(starting_band, *cap_bands)` with explicit ordering. Test that pre/post outputs match for the existing 144-test corpus.
2. F-211 — when `universe_miss_cap` archives a litigation signal, insert `operator_flags(kind='litigation_universe_miss', candidate_id=null, evidence={signal_id, profile, nos})`. Bounded by signal_id dedup so a backlog doesn't flood.
3. F-212 — change `definitive_merger_agreement → discard` to `definitive_merger_agreement → archive + operator_flag`. Operators can re-route if the signal is a false positive.
4. F-205 / F-213 — introduce `PROFILE_DIM_DEFAULTS: dict[profile, dict[dim, int]]` in dim_estimator. binary_catalyst's `approval_probability` defaults to NULL (forces resolver), not 3. short_positioning's `liquidity_tier` defaults via stricter floor (e.g., 50M).
5. F-210 — extract cap rule IDs into `RUBRIC_CAP_RULES: list[CapRule]` at top of file with name, band_effect, applies_to_profiles. The application logic reads the list. Future caps add a row, not a code branch.

**Closes:** F-205, F-206, F-210, F-211, F-212, F-213.

**Verification:** Existing 144-test rubric corpus passes with the always-append refactor. New tests for: each cap rule fires correctly, multiple-cap signals get all caps recorded, per-profile defaults apply, universe_miss flags fire.
