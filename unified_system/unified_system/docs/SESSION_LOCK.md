# Session Lock — Operational + Maintenance

**Status**: RELEASED

This file provides mutual exclusion between `unified-operational` and `unified-maintenance` scheduled tasks. They share this lock so they do not run concurrently against the same operational data.

The `unified-reporting` task does NOT use this lock — it uses its own lock at `reports/REPORTING_LOCK.md` and is always read-only to operational data.

---

## Lock protocol

1. Read this file.
2. If status is `RELEASED` — proceed to acquire.
3. If status is `HELD` — check `held_until`. If `held_until` is more than 4 hours in the past, the prior holder likely crashed — force-release and acquire. Otherwise abort with exit code 0 (don't fight for the lock; next cycle will run).
4. To acquire: atomic write this file with `HELD`, `held_by=<taskid>`, `held_since=<ISO8601>`, `held_until=<now+60min ISO8601>`.
5. At shutdown: atomic write back to `RELEASED`.
